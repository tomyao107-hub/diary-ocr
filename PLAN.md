# 日记 OCR 扩展计划（已根据决策修订）

> 修订依据：两轮专业设计问答 + 用户选择。  
> **MVP 范围：项目管理 + PDF（含页码范围）+ 递归导入文件夹；不做本地 OCR。**

## 1. 已锁定决策一览

| 维度 | 决策 |
|------|------|
| **MVP 范围** | 项目管理 + PDF；**不含**本地 OCR |
| **本地 OCR** | MVP **完全不管**，以后单独设计（不在本期预留 Backend 抽象也行，但模块拆分时避免把云端写死在 UI 里即可） |
| **长期 OCR 关系**（远期备忘） | 云端为主，本地为离线兜底；印刷体/清晰字要准，手写可较差；CPU 保守设计 |
| **部署** | 尚未决定 → 代码按「可日后打包」组织，但不做安装包 |
| **项目存储** | 统一总目录下的子文件夹（如 `~/DiaryOCRProjects/<name>/`） |
| **素材策略** | **复制进项目目录**（项目自包含） |
| **旧 session** | **可抛弃**，按新版本干净切换（不做强制迁移） |
| **导航** | 打开项目 → 校对台；**返回项目列表**（单窗口） |
| **返回列表** | **自动保存**进度与当前草稿后返回 |
| **PDF** | 整本或 **可选页码范围**；默认 **300 DPI** |
| **页模型** | 统一为「页」（图与 PDF 渲染页一视同仁） |
| **设置分层** | **几乎全全局**；项目只存进度与路径等项目态 |
| **代码结构** | **按模块拆分**；主文件 **`diary_ocr_app (5).py` → 包/模块 + `diary_ocr_app.py` 入口** |
| **测试** | 保持现有单测 + 新逻辑补测 |
| **可观测性** | 弹窗说明 + 底部日志（现状增强） |
| **删除项目** | **二次选择**：仅移除引用 **或** 彻底删除文件夹 |
| **图片侧 MVP** | **递归导入文件夹**（HEIC 不做，除非有余力） |

## 2. 目标架构（MVP）

### 2.1 用户流程

```
启动
  → ProjectHome（项目列表）
       ├─ 新建项目（名称 → 在总目录下建文件夹）
       ├─ 打开项目 → Editor（现有校对能力）
       ├─ 删除项目 → 选「仅移除引用」或「删除磁盘文件」
       └─ 全局设置（API Key / 模型 / 提示词 / 默认 DPI 等）

Editor
  ├─ 导入图片 / 递归导入文件夹（复制到项目 sources/）
  ├─ 导入 PDF（页码范围 + 300 DPI 渲染，页图落入项目）
  ├─ 单页/批量云端 OCR、校对、合并输出（逻辑对齐现版）
  └─ 返回列表（自动 session 保存）
```

### 2.2 推荐目录布局

```
%USERPROFILE%/DiaryOCRProjects/          # 全局「项目总目录」（设置可改）
  <project_slug>/
    project.json                         # id, name, created_at, updated_at
    sources/                             # 用户导入的原图、原 PDF（复制进来）
    pages/                               # 统一「页」文件（渲染结果或复制后的图）
    output/                              # 每页 .md、final_diary_output.md
    session.json                         # 页列表顺序、当前页、done、草稿
```

**页的统一约定（MVP）：**

- 列表每一项对应 `pages/` 下的一个图像文件（或稳定相对路径）。
- 从 PDF 导入：渲染为 `pages/pdfstem_p0001.jpg` 等；原 PDF 放 `sources/`。
- 从照片导入：复制到 `pages/`（或 `sources/` + `pages/` 同内容，MVP 可简化为直接进 `pages/` 并可选保留 sources 副本——**推荐：原件 sources/，工作页 pages/**，PDF 渲染只进 pages/）。

**更简 MVP 变体（若要少一层）：** 仅 `sources/` + `output/` + `session.json`，PDF 渲染页也放 `sources/rendered/`。实现时二选一，推荐 **sources / pages / output** 语义更清晰。

### 2.3 配置分层

| 层级 | 内容 | 位置 |
|------|------|------|
| 全局 | API Key、base_url、model、temperature、max_tokens、user/system prompt、**项目总目录**、默认 PDF DPI、批量并发 | `~/.diary_ocr_config.json`（可沿用并扩展字段） |
| 项目 | 名称、页路径列表顺序、current、done、drafts、（可选）备注 | `project.json` + `session.json` |

项目 **不** 存 API Key；换机器可拷贝项目文件夹，Key 仍在本机全局配置。

### 2.4 模块拆分建议

```
diary_ocr/
  __init__.py
  app.py                 # QApplication、启动 ProjectHome
  paths.py               # 总目录、规范路径
  project_store.py       # 列表/新建/打开/删除
  models.py              # ProjectMeta、Session 结构
  session.py             # 现 SessionManager 迁入，按项目路径
  pdf_import.py          # 页码范围、DPI、渲染到 pages/
  image_import.py        # 复制、递归文件夹、格式过滤
  image_compress.py      # ImageCompressor
  ocr_client.py          # 现 OCRAPIClient（云端）
  workers.py             # OCR/Batch/Preview/Connection workers
  ui/
    project_home.py
    main_window.py       # 校对台
    settings_dialog.py
    widgets.py           # ZoomableImageViewer, LogConsole
  style.py               # APP_STYLE
diary_ocr_app.py         # 薄入口：python diary_ocr_app.py
run_diary_ocr.cmd        # 指向新入口
tests/
  test_core.py           # 适配新导入路径
  test_project_store.py
  test_pdf_import.py
```

依赖新增（MVP）：**PyMuPDF**（`pymupdf`）。其余保持 PyQt6 / Pillow / openai / python-dotenv。

## 3. MVP 功能规格

### 3.1 项目管理

- 启动默认 `ProjectHome`：卡片或列表展示名称、页数、更新时间。
- **新建**：输入名称 → slug 目录（重名则后缀 `_2`）→ 写 `project.json` → 可直接打开。
- **打开**：加载 `session.json` + 校验页文件是否存在；缺页从列表剔除并打日志（对齐现 materialize 思路）。
- **返回列表**：flush 草稿 → 写 session → 切回 Home（销毁或隐藏 Editor，推荐隐藏/替换 central 或关 Editor 回 Home，**单窗口**）。
- **删除**：对话框二选一  
  1. 仅从索引移除（若使用全局 index）；若「总目录扫描即列表」则「移除引用」意义较弱——见下。  
  2. 删除整个项目文件夹（需确认输入项目名或二次确认）。

**项目发现策略（推荐）：**  
不单独维护易漂的 index 为主数据源，而是 **扫描总目录下含 `project.json` 的子文件夹** 生成列表；删除「仅移除引用」可改为「从列表隐藏」需 hidden 标记，或 MVP **只提供彻底删除** + 用户手动挪走文件夹。  

结合你已选「二次选择」：

- **彻底删除**：`shutil.rmtree(project_dir)`  
- **仅移除引用**：写入总目录级 `projects_index.json` 的 `archived`/`removed` 集合，扫描时跳过；磁盘保留。  

### 3.2 PDF 导入

- 菜单/工具栏：「导入 PDF」。
- 选文件 → 对话框：**起始页、结束页**（默认 1–N）、展示总页数；DPI 用全局默认 300（设置可改，MVP 可不做单次覆盖）。
- 将原 PDF **复制**到 `sources/`。
- 后台线程渲染选中页 → `pages/`，进度条 + 可取消（尽量）。
- 渲染完成后加入会话页列表（队尾或按自然序），不自动开始 OCR。
- 错误：坏 PDF、页码越界、磁盘满 → 弹窗 + 日志。

### 3.3 图片 / 文件夹

- 现有多选导入：改为 **复制进项目** 再入列表（不再只引用外部路径）。
- **递归导入文件夹**：遍历子目录，过滤扩展名，复制后入列表；跳过隐藏/无权限并记日志。
- 支持扩展名：保持现有集合（暂无 HEIC）。

### 3.4 校对台（继承并约束）

- 保留：单页 OCR、批量 OCR、合并、缩放预览、自然排序/拖拽、保存并下一页等。
- 所有输出路径改为项目 `output/`。
- 去掉或弱化「全局恢复上次」：改为 Home 显示最近项目 + 自动打开可选（MVP 可不做自动打开）。
- 全局 session 文件 **不再作为主存储**；可停止写入或忽略旧文件。

### 3.5 明确不做（MVP）

- 本地 CPU OCR / 引擎切换 UI  
- HEIC  
- 多窗口多项目并行  
- 项目级提示词/模型覆盖  
- 旧 session 一键迁移  
- 安装包  
- PDF 密码、导出 DOCX/PDF 成册增强

## 4. 版本安排与实现分期（仅 MVP 内）

### 4.1 发布原则

- 使用语义化版本 `MAJOR.MINOR.PATCH`；MVP 开发期从 `v0.1.0` 起，完整验收后发布 `v1.0.0`。
- `MINOR` 只承载一个可独立验收的能力增量；`PATCH` 只修复缺陷，不追加范围。
- 应用版本与数据格式版本分离：应用版本集中定义；`project.json` 与 `session.json` 各自维护 `schema_version`。
- `v0.1.0` 前的全局旧 session 不承诺迁移；从 `v0.2.0` 起，已经创建的项目数据必须向后兼容，必要时提供自动迁移。
- 每个版本必须满足：新增逻辑有测试、现有回归测试全绿、启动脚本可用、验收项通过，才创建 `vX.Y.Z` 标签。
- `main` 始终保持可启动；未达到版本出口条件的功能不进入发布标签。

### 4.3 版本与原 Phase 对应

#### v0.1.0 — 工程基线（Phase A 的拆分前置）

1. 建立 `diary_ocr/` 包结构与 `diary_ocr_app.py` 新入口。
2. 按模块搬迁现有代码，先不改变存储和 UI 行为。
3. 适配现有 8 项测试的导入路径并保持全绿。
4. 更新 `run_diary_ocr.cmd`。

#### v0.2.0 — Phase A：骨架与项目存储

1. `project_store`：扫描、新建、删除两种模式。  
2. `ProjectHome` UI + 打开/返回导航。  
3. 将 `SessionManager` 绑定到项目 `session.json`。  
4. 新增配置项 `projects_root`。  

**验收：** 可建两个空项目，开关互不干扰；返回自动保存。

#### v0.3.0 — Phase B：导入复制语义 + 递归文件夹

1. 图片导入改为 copy-into-project。  
2. 递归文件夹导入。  
3. session 存相对项目根目录的路径，避免盘符迁移脆弱。
4. 明确定义重名文件、重复导入、无权限文件和不支持格式的处理方式。

**验收：** 外部删除原图不影响项目内页；子文件夹图片能进列表。

#### v0.4.0 — Phase C：PDF

1. `pdf_import` + 页码范围 UI。  
2. 默认 300 DPI，逐页渲染进 `pages/`。  
3. 增加进度、取消、磁盘写入失败和损坏 PDF 的错误处理。  
4. 与批量 OCR/合并回归。  

**验收：** 指定 2–5 页导入，顺序正确，OCR/合并按列表。

#### v0.5.0 ～ v1.0.0 — Phase D：硬化与稳定发布

1. 单测：project_store、PDF 页范围、复制导入、session 相对路径。  
2. 保持现有 8 项回归测试通过。  
3. 增加项目完整流程的集成测试或可重复人工验收脚本。  
4. 梳理日志文案与错误弹窗。  
5. 更新 `run_diary_ocr.cmd`、requirements 和发布说明。  
6. `v0.5.0` 后冻结功能；`v1.0.0` 只处理阻塞发布的缺陷。

### 4.4 MVP 之后的版本边界

- `v1.0.x`：仅稳定性、兼容性和安全修复。
- `v1.1.0+`：从远期备忘中按价值单独选题；安装包、HEIC、项目级 prompt 等不得捆绑进入一个版本。
- `v2.0.0` 候选：本地 OCR/引擎切换。该能力涉及依赖体积、模型选型和数据格式变化，需另立设计计划，不提前承诺日期。

## 5. 风险与设计对策

| 风险 | 对策 |
|------|------|
| 单文件拆分回归 | 先拆再改行为；每 Phase 跑单测 |
| 复制导致磁盘膨胀 | 状态栏提示项目体积；删除项目可彻底删 |
| 300 DPI 大 PDF 内存 | 逐页渲染写盘，不在内存攒全页；限制并发 1 |
| 路径绝对路径搬家即挂 | session 存项目相对路径 |
| 云端逻辑与 UI 缠在一起 | Worker 仍吃 client 接口；不在本期做本地，但 `ocr_client` 独立模块便于远期替换 |
| 用户误删项目 | 彻底删除二次确认（确认名或「删除」字样） |

## 11. v1.0 → v2.0 升级路线

### 11.1 总体目标

v1.0 之后不再以一次性大重构推进，而是让每个 `MINOR` 版本只承担一个可以独立验收的产品主题。总体演进方向如下：

| 阶段 | 产品定位 | 核心成果 |
|------|----------|----------|
| **v1.0.x** | 稳定的项目化云端 OCR 工具 | 修复真实使用缺陷，保证数据安全 |
| **v1.1.0** | 可靠批量处理 | 可恢复任务队列、失败重试和处理报告 |
| **v1.2.0** | 高质量素材入口 | 去重、HEIC、方向/裁边/图像增强 |
| **v1.3.0** | 可交付的数字化成果 | 章节、DOCX/PDF/Markdown 成册导出 |
| **v1.4.0** | 可安全分发的桌面产品 | 安装包、凭据保护、备份与更新 |
| **v1.5.0** | 多引擎架构准备 | 统一 OCR 引擎接口和固定评测体系 |
| **v2.0.0** | 本地、云端、混合 OCR 平台 | 离线 OCR、引擎切换和混合决策 |

路线原则：

1. `v1.x` 始终保持现有云端 OCR 可用，不因下一阶段重构中断主流程。
2. 功能、数据迁移和测试在同一版本交付，不允许“先改格式、以后补迁移”。
3. 本地 OCR 只在 v1.5 完成引擎解耦后进入 v2.0，避免 UI、队列和引擎同时重写。
4. 每个版本通过出口条件后才能开始下一版本；时间可以顺延，质量门槛不压缩。

### 11.2 v1.0.x — 稳定性维护

**目标：** 用真实项目验证 v1.0，不引入新的产品范围。

范围：

- 修复崩溃、路径、PDF 导入、恢复和合并问题。
- 覆盖中文用户名、中文路径、长路径和不同磁盘位置。
- 改善大图、大 PDF、损坏文件、磁盘不足和权限不足处理。
- 梳理用户可理解的错误文案和诊断日志。
- 保持 `project.json` 与 `session.json` schema 不变。

建议发布节奏：

- `v1.0.1`：首轮真实使用发现的阻塞缺陷。
- `v1.0.2`：兼容性、数据安全和性能问题。
- `v1.0.3+`：仅在确有缺陷时发布，不为版本号增加功能。

出口条件：

- 无已知数据丢失或项目损坏问题。
- 50–100 页项目连续导入、OCR、校对、合并和恢复通过。
- 旧项目在所有 v1.0.x 版本之间可以直接打开。

### 11.3 v1.1.0 — 可靠批量 OCR

**主题：** 任务可恢复，失败可诊断、可重试。

功能范围：

- 持久化 OCR 任务队列。
- 页面任务状态：`pending`、`running`、`succeeded`、`failed`、`cancelled`。
- 支持“处理全部”“仅处理未完成”“仅重试失败”。
- 超时、HTTP 429 和服务端 5xx 使用有上限的指数退避重试。
- 单页永久失败不阻塞其余页面。
- 用户取消后保留已成功结果和未完成队列。
- 程序异常退出后，将遗留的 `running` 自动恢复为 `pending`。
- 保存每页尝试次数、最后错误和完成时间。
- 批处理结束生成成功、失败、跳过和取消摘要。

建议的会话数据：

```json
{
  "schema_version": 2,
  "pages": [
    {
      "path": "pages/page001.jpg",
      "status": "succeeded",
      "attempts": 1,
      "last_error": null,
      "completed_at": "2026-07-18T12:00:00+08:00",
      "output": "output/page001.md"
    }
  ]
}
```

迁移：

- `session.json` schema 1 → schema 2。
- 迁移前生成 `session.json.bak`。
- 已存在对应 Markdown 的页面初始化为 `succeeded`，其余初始化为 `pending`。
- 迁移必须幂等；重复启动不得重复改变状态或丢失草稿。

出口条件：

- 100 页队列可以取消、关闭程序、重启并继续。
- 已成功页面不会被无意重复识别或覆盖。
- 429、超时、5xx 和单页永久失败模拟测试通过。
- 失败报告可以定位到具体页面和最后错误。

**预计工作量：** 3–4 周。

### 11.4 v1.2.0 — 素材质量与导入增强

**主题：** 输入素材可追踪、可去重、可无损重新处理。

功能范围：

- HEIC/HEIF 导入。
- 使用 SHA-256 检测重复原件，导入时明确提示跳过或保留副本。
- 导入结果报告：成功、重复、跳过、不支持和失败。
- EXIF 方向修正。
- 页面旋转、裁边、透视矫正、灰度和对比度增强。
- PDF 页面缩略图和页码范围预览。
- 缺页、重复页、异常尺寸和异常方向提示。
- 原件始终保留在 `sources/`；处理后的工作页写入 `pages/`。
- 可根据处理参数从原件重新生成工作页。

建议的页面模型：

```json
{
  "id": "page-uuid",
  "path": "pages/page001.jpg",
  "source": "sources/photo001.heic",
  "sha256": "...",
  "rotation": 90,
  "preprocessing": {
    "crop": null,
    "perspective": null,
    "grayscale": false,
    "contrast": 1.0
  }
}
```

迁移：

- `project.json` schema 1 → schema 2。
- 旧项目的页面路径转换为页面对象。
- 无法确定原件来源时，`source` 可以为空，但现有工作页必须继续可用。

出口条件：

- 删除外部文件后项目仍完全可用。
- 重复导入行为明确，不产生静默覆盖。
- 所有图像处理均可撤销或从原件重新生成。
- HEIC、手机照片和倾斜扫描件完整走通导入与 OCR。

**预计工作量：** 3–4 周。

### 11.5 v1.3.0 — 成册与交付

**主题：** 将逐页转录结果组织成可以交付的完整文档。

功能范围：

- 项目章节、页面标题、日期和备注。
- 按章节合并，支持逻辑页码和来源页码。
- Markdown 完整导出。
- DOCX 成册导出。
- PDF 成册导出。
- 原图与转录文本对照版。
- 封面、目录、页眉、页脚和页码。
- 可复用的导出配置模板。
- 导出前检查未完成、缺失或失败页面。

输出结构：

```text
output/
  pages/
  exports/
    diary.md
    diary.docx
    diary.pdf
  export_profile.json
```

数据约束：

- `project.json` schema 2 → schema 3，增加章节、标题、逻辑页码和导出配置。
- 单页校对文本继续独立保存，不将 DOCX/PDF 排版信息写回原始转录。
- 导出是可重复生成的派生结果，失败时不得破坏单页文本。

出口条件：

- Markdown、DOCX 和 PDF 的页面顺序与章节一致。
- 200 页项目可以稳定导出。
- 同一项目可使用不同模板重复导出。
- 缺页和未完成页在导出前有明确提示。

**预计工作量：** 4–5 周。

### 11.6 v1.4.0 — 分发、安全与备份

**主题：** 让非开发用户能够安装、升级、备份和恢复。

功能范围：

- Windows 安装包和便携版。
- 开始菜单、桌面快捷方式及环境检测。
- API Key 改存 Windows 凭据管理器，不再明文写入 JSON。
- 全局配置导入与导出。
- 项目备份、恢复和归档包。
- 脱敏诊断包与崩溃报告。
- Stable/Beta 更新通道。
- 可选自动更新及失败回退。

安全规则：

- API Key 不写入项目、项目备份和诊断包。
- 诊断日志默认不包含 OCR 正文和图片内容。
- 恢复归档前校验路径，禁止覆盖项目目录之外的文件。
- 应用升级和卸载不得删除用户项目。

出口条件：

- 全新 Windows 环境无需自行安装 Python 即可运行。
- 从上一稳定版升级后项目和设置保持可用。
- 项目备份可在另一台电脑恢复。
- 更新失败能够回退到上一个可运行版本。

**预计工作量：** 4–6 周。

### 11.7 v1.5.0 — 多引擎架构准备

**主题：** 在不改变用户默认行为的前提下，完成 OCR 引擎解耦和评测基础。

统一接口建议：

```python
class OCREngine:
    def capabilities(self) -> EngineCapabilities:
        ...

    def recognize(
        self,
        image: bytes,
        options: OCROptions,
        cancel_token: CancelToken,
    ) -> OCRResult:
        ...
```

统一结果建议：

```json
{
  "text": "...",
  "engine": "qwen-cloud",
  "engine_version": "...",
  "model": "...",
  "duration_ms": 2180,
  "warnings": [],
  "created_at": "...",
  "parameters": {}
}
```

工程范围：

- 将现有云端 OCR 迁入统一 `OCREngine` 接口。
- UI 和任务队列不再直接依赖 OpenAI 兼容客户端。
- 增加引擎能力声明：联网、语言、手写、印刷体、CPU/GPU 等。
- OCR 结果记录引擎、模型、版本、参数和耗时。
- 增加离线模拟引擎，用于无网络测试和故障注入。
- 建立脱敏金标准数据集和可重复评测工具。
- 评测字符错误率、页成功率、处理时间、峰值内存和磁盘占用。

本版本仍只向普通用户提供稳定的云端引擎，不发布未达标的本地 OCR。

出口条件：

- 现有云端功能在新接口上行为一致。
- 更换模拟引擎无需修改 UI 和任务队列。
- 每次 OCR 结果都能追踪其引擎、模型和参数。
- 固定评测集可以生成可比较的基准报告。

**预计工作量：** 3–4 周。

### 11.8 v2.0.0 — 本地、云端与混合 OCR

**重大版本目标：** 从单一云端 OCR 工具升级为具有统一任务、结果和质量模型的多引擎 OCR 平台。

#### 本地模式

- 图片不离开电脑，无网络也可以识别。
- 优先支持 CPU，允许可选 GPU 加速。
- 显示模型体积、预计速度、语言和字体能力。
- 支持模型下载、完整性校验、升级和删除。
- 模型不可用时给出明确诊断，不静默切换云端。

#### 云端模式

- 保留现有云端 OCR 能力。
- 支持多个 OpenAI 兼容提供商。
- 每个提供商独立保存连接、模型、限流和重试设置。
- 支持连接测试和项目级云端禁用。

#### 混合模式

推荐流程：

1. 先执行本地 OCR。
2. 根据结果完整性、规则或人工标记判断疑难页面。
3. 疑难页面加入云端候选队列。
4. 用户确认后才发送云端。
5. 同时保留本地和云端结果，允许对比并选择。

策略示例：

- 清晰印刷体优先本地。
- 手写、褪色、破损和复杂版式优先云端。
- 无网络时保留云端候选任务，联网后由用户继续。
- 隐私项目可以强制本地，禁止任何网络发送。
- 单页策略可以覆盖项目默认策略。

v2.0 功能范围：

- OCR 引擎管理器。
- 本地模型生命周期管理。
- CPU/GPU 能力和可用内存检测。
- 云端提供商管理。
- 全局、项目级和页面级 OCR 策略。
- 本地与云端结果并排比较。
- 重新识别时保留历史结果。
- OCR 来源、参数和时间完整追踪。
- 隐私模式。
- 引擎性能与质量评测面板。

明确不进入 v2.0：

- 自训练或微调模型。
- 在线协作与多用户权限。
- 云端项目同步。
- 移动端。
- 自动翻译。
- 面向机构的大规模档案管理。

这些能力在 v2.0 稳定后按独立的 `v2.x` 主题评估。

出口条件：

- 本地、云端和混合三种模式均可完整执行“导入→识别→校对→合并→恢复”。
- 本地模式在断网环境下可用且不产生网络请求。
- 混合模式未经用户确认不会上传候选页面。
- 重新识别不覆盖人工校对文本。
- 引擎或模型不可用不会破坏项目状态。
- v1.x 项目可以自动迁移并保持全部原始素材、转录和导出结果。

**预计工作量：** Alpha、Beta 到 Stable 共 8–12 周。

### 11.9 数据版本与兼容策略

应用版本与数据格式继续分别管理：

- 应用版本：`MAJOR.MINOR.PATCH`
- `project.json`：`schema_version`
- `session.json`：`schema_version`
- OCR 结果：独立 `result_schema_version`
- 备份归档：独立 `archive_schema_version`

建议演进：

| 应用版本 | 主要数据变化 |
|----------|--------------|
| v1.0.x | 不改变 schema |
| v1.1 | 会话增加持久化任务状态 |
| v1.2 | 页面元数据、来源与预处理参数 |
| v1.3 | 章节和导出配置 |
| v1.4 | 原则上不改变核心项目格式 |
| v1.5 | OCR 引擎和结果来源信息 |
| v2.0 | 多结果历史和项目级引擎策略 |

迁移规则：

1. 每次迁移前生成带时间戳的备份。
2. 迁移必须幂等，重复运行得到相同结果。
3. 新版本必须能读取所有受支持的旧 schema。
4. 旧版本遇到更高 schema 时只读打开或明确拒绝，绝不覆盖。
5. 迁移失败时保留原文件，并提供可复制的错误报告。
6. 原始素材、OCR 原始结果、人工校对文本和导出文件分层保存。
7. 重新 OCR 不得覆盖人工校对结果。
8. 项目内路径始终使用相对项目根的规范路径。

### 11.10 测试与发布质量门槛

每个版本必须满足：

- 所有旧回归测试通过。
- 新增逻辑有单元测试，关键流程有集成测试。
- 至少完成一次从上一稳定版升级的数据迁移测试。
- 迁移失败不会破坏原项目。
- 启动脚本或安装包在干净环境验证通过。
- README、Changelog、schema 说明和已知限制同步更新。
- 无 P0/P1 缺陷。
- 连续两次完整回归通过后才创建正式标签。

长期维护一个脱敏测试集，至少覆盖：

- 清晰印刷体。
- 横排和竖排手写。
- 简体、繁体和历史异体字。
- 褪色墨迹和破损纸张。
- 倾斜、阴影和透视变形的手机照片。
- 重复页、缺页和错误方向。
- 50–200 页 PDF。

关键质量指标：

| 维度 | 指标 |
|------|------|
| 数据安全 | 项目损坏率、迁移失败恢复率 |
| OCR 可靠性 | 页成功率、重试后成功率、任务恢复率 |
| OCR 质量 | 字符错误率、疑难页人工修订量 |
| 性能 | 单页耗时、峰值内存、大项目启动时间 |
| 可交付性 | 导出成功率、页序/章节一致性 |
| 隐私 | 未授权网络请求数必须为 0 |
